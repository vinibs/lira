Atenção: ao iniciar um projeto utilizando o Lira, é preciso ajustar a tag <base> no arquivo index.html alterando seu valor para indicar a raiz do projeto, sempre iniciando e terminando com "/". Download e instalação

Para baixar o Lira, é preciso acessar a página no GitHub e realizar o download do .zip contendo toda a estrutura. Então, basta descompactar os arquivos e configurar o caminho base.


Atualização do framework

Quando for necessário atualizar o framework, a forma mais prática de fazê-lo, sem que haja perda ou sobrescrita dos dados ou configurações já realizados no projeto, é baixar o arquivo app.js novamente e substituí-lo na raiz do projeto. Entretanto, é importante sempre observar se não houve alterações na estrutura básica de arquivos.



Estrutura básica

Uma vez colocados os arquivos base do sistema em um diretório para iniciar o desenvolvimento, basta montar suas páginas com seus arquivos HTML, JavaScript e CSS, colocando-os em seus lugares dentro do sistema para começar a criar um pequeno sistema em página única maravilhoso.

A estrutura básica não é muito grande ou complexa. Ainda assim, merece atenção para que cada componente esteja em seu devido lugar durante o desenvolvimento da aplicação.

A relação dos diretórios padrão é a seguinte, podendo diferir dos de diretórios e arquivos baixados do repositório, mas contendo o essencial e ideal para a execução do sistema:

  • / (raiz da aplicação)
    • src   // Estrutura básica do sistema
      • app   // Contém os arquivos executados na aplicação
        • global
        • main.js   // Código JS global, carregado por padrão em todas as páginas
        • pages
          • home   // Arquivos executados como a página inicial do sistema
            • home.page.css   // Estilo destinado unicamente a esta página
            • home.page.html   // HTML incluso no corpo do documento
            • home.page.js   // Configurações de qual HTML, CSS e funções JS devem ser executados
        • app.body.html   // HTML carregado em todas as páginas do sistema, útil para layouts
        • app.routes.js   // Arquivo onde são registradas as rotas da aplicação
      • assets   // Destinado a arquivos como imagens, fontes ou bibliotecas JS
        • img
          • lira.png
    • .htaccess
    • app.js   // Arquivo JavaScript contendo toda a base do sistema
    • favicon.png   // Ícone aplicado por padrão a todas as páginas
    • index.html   // Arquivo HTML que receberá todas as requisições
    • LICENSE
    • main.css   // Arquivo de estilos global, aplicado a todas as páginas
    • robots.txt

Funcionamento básico

O Lira possui um funcionamento relativamente simples. Como pode ser observado na seção de estrutura básica, há diretórios específicos para o armazenamento das páginas que irão compor o código completo da tela do sistema. O diretório "pages" espera que haja um diretório específico para cada página do sistema e, dentro de cada um destes, espera-se que haja ao menos um arquivo .page.js e um .page.html.

O arquivo JavaScript é responsável por indicar ao roteador qual deverá ser o HTML carregado na página, bem como qual arquivo CSS (se houver) será carregado exclusivamente para essa tela, alguns atributos da página, como variáveis, o título da página e código JavaScript a ser executado ao renderizar a tela. Veja em mais detalhes na seção arquivos de setup.

Já o arquivo HTML é responsável por fornecer ao JS todo o código específico a ser renderizado na tela. Ao contrário de alguns frameworks JavaScript, o Lira não fornece funções embutidas ao HTML para processamento, portanto funcionalidades como condições e loops são realizados no handler definido no arquivo JS. Mais informações estão disponíveis na seção [ALGUM NOME].

Para carregar estes dados e renderizá-los no navegador, existe o roteador interno. Ele realiza simples procedimentos para carregar os componentes da página. São eles:

  • Ler as rotas do arquivo app.routes.js e verificar se alguma corresponde à URL da requisição atual
  • Buscar, dentro do diretório /src/app/pages, pelo arquivo que tenha a extensão ".page.js", seguindo a hierarquia de diretórios e o nome do arquivo presentes no atributo "setup" da rota.
  • Obter as informações sobre qual HTML e CSS usar, quais comandos devem ser executados ao renderizar a página
  • Carregar o HTML do arquivo dentro do layout definido em app.body.html, inserir o CSS e executar o código JS definido, caso hajam estes dois últimos.

Veja mais sobre a preparação do roteamento em rotas.



Configuração inicial

Para iniciar um projeto com o Lira é necessário realizar uma pequena configuração inicial. Após extrair os arquivos baixados e colocá-los em um diretório no seu servidor, deve-se configurar o caminho que identifica a raiz da aplicação.

Para isso, é preciso abrir o arquivo index.html e editar a tag <base>, adicionando à propriedade href o caminho referente ao diretório raiz da sua aplicação a partir da raiz do servidor. Por exemplo, se o Lira compuser diretamente a raiz do servidor, o caminho fica apenas "/". Entretanto, se, na raiz do servidor houver um diretório chamado, por exemplo, de "spa", o caminho seria "/spa/".

Também é importante ter em mente que, para que a plicação funcione corretamente, incluindo o roteamento e as demais funcionalidades, é preciso que o projeto esteja rodando sobre um servidor HTTP, e que haja um arquivo de configuração que redirecione todas as requisições ao arquivo index.html, como é o caso do arquivo .htaccess incluído por padrão na estrutura, que realiza esta função no servidor Apache.

Observe que todo e qualquer caminho aplicado à tag <base>, por se tratar de uma tag do próprio padrão HTML, deve ser iniciada com "/" E finalizada com uma outra "/" quando fizer referência a um caminho interno e, como, por definição, aqui tratamos apenas de caminhos internos do servidor, é necessário que o atributo href contenha um valor neste padrão. Caso contrário, a aplicação não funcionará corretamente.



Rotas

O Lira possui um sistema interno de roteamento, em que a cada requisição é buscado no arquivo app.routes.js um objeto que possua o valor path equivalente à URL da requisição atual.


Criação de uma rota

Para definir uma rota, é necessário editar o arquivo app.routes.js, localizado no diretório src/app. Neste arquivo há a definição de um vetor de objetos. Uma rota é um objeto, dentro deste vetor, que contenha dois atributos: path e setup.

O atributo path representa o caminho de URL que será identificado. Este caminho pode conter parâmetros variáveis ou não. Para mais informações, veja rota com parâmetros.

Este caminho deve ser referente à raiz da aplicação. Pode tanto ser iniciado por uma "/" ou não, bem como ser também finalizado por uma barra ou não. Ou seja, uma rota "pagina1/pagina2" é equivalente a "/pagina1/pagina2", "pagina1/pagina2/", "/pagina1/pagina2/".

Há também, para o atributo path, um "operador coringa", que pode ser usado para indicar qualquer rota que não pertença às rotas declaradas anteriormente. Tal operador é o caractere "*". Quando utilizado, deve ser escrito como última rota do arquivo ou, caso contrário, as rotas que estiverem após esta, no vetor, não serão reconhecidas.

O atributo setup, por sua vez, espera um caminho, a partir do diretório src/app/pages, para um arquivo de setup com a extensão .page.js. Este arquivo deve ser referenciado sem a extensão neste atributo e a hierarquia de diretórios é demarcada pelo caractere "/". Por exemplo, se houver um arquivo cujo caminho seja src/app/pages/funcionario/funcionario.page.js, o atributo setup deve conter apenas o caminho funcionario/funcionario (que também pode ou não iniciar ou encerrar com uma barra).

Veja um exemplo de objeto de rota, em que o caminho da rota é "/pessoas/lista" e o setup está localizado em "src/app/pages/pessoas/lista.page.js": 

{
   path: 'pessoas/lista', // Identifica o caminho da rota, a URL
   setup: 'pessoas/lista' // Identifica que deve ser carregado o arquivo lista.page.js de dentro do diretório src/app/pages/pessoas
}

Um outro exemplo, utilizando o caractere coringa para definir uma página de erro 404: 

{
   path: '*', // Indica, pelo operador coringa, que qualquer rota que não estja entre as anteriores deverá chamar a página de erro 404
   setup: 'erros/404' // Identifica que deve ser carregado o arquivo 404.page.js de dentro do diretório src/app/pages/erros
}

Nota: observe que o atributo path quando deixado com uma string vazia ("") é equivalente à rota "/", ou seja, se refere à raiz da aplicação. Se algum dos atributos da rota não for definido, o roteador não funcionará corretamente.


Rota com parâmetros

Também é possível definir rotas que esperam parâmetros (variáveis) através da URL. Por exemplo, caso seja necessário passar um código identificador para ser enviado em uma requisição AJAX, é possível que ele seja passado através da URL utilizando a notação "{nome_var}", onde "nome_var" representa o nome que identificará a variável no código JavaScript da página.

Por exemplo, se uma rota "cliente" esperar um identificador para fazer a requisição ao servidor dos dados de um cliente específico, pode-se definir o path da rota como "cliente/{cod}". Desta forma, será enviada à função handler do arquivo definido em setup uma variável como parâmetro. Isso implica que tal função deve esperar um argumento em sua declaração.

Como parâmetro é passado um objeto em que seus atributos têm como nome o texto especificado na rota (no caso do exemplo, "cod") e como valor o dado passado através da URL de acesso à página.

Um exemplo de uso de uma rota com parâmetros: 

{
   path: 'cliente/{id}/ver', // Indica que será passado um atributo chamado "id" com o valor dado na URL, nesta mesma posição da string, no objeto enviado como parâmetro para a função handler do setup
   setup: 'cliente/ver' 
}

Também é possível passar mais de um parâmetro na rota, como por exemplo em "section/{attr1}/{attr2}/file/{attr3}.

Nota: observe que é preciso definir, na função handler do arquivo de setup, um atributo a ser passado na declaração da função, para que o dado obtido possa ser utilizado. Veja mais em arquivos de setup.



Arquivos de setup

Arquivos de setup são arquivos JavaScript que especificam ao roteador que HTML deve ser carregado, o título da página, quais variáveis devem ser acessíveis durante a execução da página, qual processamento JS deve ser executado ao carregar a tela e qual arquivo de estilo específico para aquela página deve ser carregado ao ser identificada uma determinada rota.

Em outras palavras, o arquivo de setup referenciado por uma rota indica ao roteador o que deve ser carregado na tela e o que deve ser executado ao carregar a página quando esta rota for acessada por um usuário. Pode-se dizer também que é o arquivo que faz a mágica acontecer.


Criação de um setup

Para criar um arquivo de setup, deve-se criar um arquivo JavaScript com a extensão ".page.js" dentro de um diretório específico para a página (ou grupo de páginas com alguma característica em comum), dentro do diretório "src/app/pages". Essa organização é importante para a manutenção do sistema e também para que o roteador funcione corretamente. Veja mais em organização de views.

Um arquivo de setup basicamente é uma atribuição de um objeto JSON a uma variável JavaScript já existente chamada "page". O objeto deve ter, obrigatóriamente, os atributos "title", que representa o título carregado na janela do navegador do cliente, e pageFile, que indica o caminho, a partir da localização do arquivo atual, para o arquivo HTML contendo o conteúdo da página.

Outros atributos são possíveis de serem utilizados. São eles o atributo styleFile, que, assim como "pageFile", indica um caminho a partir do diretório do arquivo atual para um arquivo CSS que será carregado apenas quando esta página for visitada, attr, que recebe atributos (valores, variáveis) que ficam acessíveis tanto ao HTML - com códigos JS inline, como "onclick", por exemplo - quanto à função handler, outro atributo possível, que espera uma função anônima contendo todo o conteúdo JavaScript que deve ser executado nesta página, sejam códigos estruturais ou funções que podem inclusive ser chamadas por elementos HTML com código JS inline.

Para mais informações sobre as funções pré-existentes para uso na função handler, veja a seção funções para a handler.

Um exemplo de um arquivo de setup encontra-se abaixo. 

src/app/pages/main/home.page.js:

			
page = {

            
   title: 'Jony\'s Blog - Home', // Define o título da página

			
   pageFile: 'home.page.html', // Indica que deve ser carregado o arquivo home.page.html

			
}

		
        

            Veja também um novo exemplo, contendo um modelo de aplicação de todos os atributos possíveis. Note que o
            arquivo se encontra em um diretório específico para os arquivos referentes a esta página, e, portanto, um
            diretório distinto do usado no exemplo anterior.
        

        
            
src/app/pages/works/works.page.js:

            

			
page = {

            
   title: 'Jony\'s Blog - Works',

			
   pageFile: 'works.page.html',

			
   styleFile: 'works.page.css', // Indica que deve ser carregado o estilo works.page.css

            
   attr: { // Passa um objeto cujos atributos podem ser acessados pela handler ou pelo HTML

            
       workType: 1 // Define um atributo qualquer desse objeto

            
   },

            
   handler: function () { // Define o código a ser executado durante a execução da rota e exibição da página

            
       alert(page.attr.workType); // Exibe o valor do atributo workType da página, definido no atributo attr deste mesmo arquivo

            
       // Output: "1"

            
           

            
       // Any JS Code here

            
   } 

			
}

		
        

            Nota: qualquer atributo definido em attr é invocado através da chamada
            page.attr.[nome_atributo], onde "[nome_atributo]" representa o nome dado ao atributo, que,
            no caso do exemplo, é workType.
        


        

        
Handler com parâmetros

        

            Quando se define uma rota com parâmetros é preciso indicar à função
            handler de um arquivo de setup que espera-se receber um parâmetro. Para isso, basta, na declaração
            da função, adicionar uma variável qualquer como atributo. Um nome adequado pode ser "get", visto
            que são dados vindos de uma requisição GET. Entretanto, o nome da variável fica a cargo do desenvolvedor.
        

        

            Dentro da função handler, por sua vez, para acessar os dados vindos por parâmetro get, basta acessar,
            através do nome dado ao atributo na rota (como "{id}", por exemplo), o atributo da variável passada como
            parâmetro para a função.
        

        

            Por exemplo, se o parâmetro se chamar "get" e o atributo definido na rota se chamar "id",
            seu valor é acessível, dentro da função handler, por get.id.
        

        

            Veja um exemplo de uso de uma rota com parâmetros em conjunto com um handler com parâmetros, assumindo
            que são passados dois atributos pela URL: "id" e "nome":
        

        
            
Route attributes: id, nome

            

			
page = {

            
   ...

            
   handler: function (get) { // Recebe a o parâmetro com os valores

            
       alert(get.id); // Exibe o valor do id passado pela rota

            
       alert(get.nome); // Exibe o valor do nome passado pela rota

            
   } 

			
}

		

        



        
Views das páginas

        

            A view de uma página é feita, primordialmente, por um arquivo HTML contendo o conteúdo que deve ser
            renderizado e exibido. Este arquivo deve estar no diretório src/app/pages e,
            preferencialmente, em um diretório específico para a tela em questão, junto ao setup que renderizará
            este arquivo. Além disso, também é possível definir um arquivo CSS que será aplicado individualmente à
            página, ou seja, propriedades que não se propagarão para outras telas. Entretanto, conforme explicitado
            em arquivos de setup, tanto o arquivo HTML quanto o CSS, se houver, devem ser
            indicados no arquivo JS que gerencia a página.
        

        

            É recomendável que o arquivo HTML conte com a extensão ".page.html", enquanto que o arquivo CSS
            tenha a extensão ".page.css" e que seu nome faça referência à página ou rota a que se aplica.
            Assim, um arquivo cliente.page.js estaria em conjunto com os arquivos cliente.page.html e
            cliente.page.css, dentro de um diretório "cliente". Isso por uma questão de organização.
            Entretanto, pode-se colocar qualquer nome e pode-se ou não utilizar ".page" nos nomes dos arquivos,
            desde que a referência dentro do arquivo de setup a estes arquivos esteja correta.
        


        

        
Template HTML

        

            É possível também criar um template em HTML para ser aplicado a todas as páginas. Entretanto, é possível
            apenas gerar um único template.
        

        

            Para isso, basta construir o template com as estruturas comuns, como cabeçalhos, rodapés e menus no arquivo
            src/app/app.body.html, mantendo, obrigatóriamente, o elemento
            <div id="route-content"></div>, que é o responsável por incluir os conteúdos das demais
            páginas na tela.
        

        

            Este elemento é referente ao real corpo das páginas, portanto é importante que ele seja utilizado sempre
            como um container do corpo, não como cabeçalho, rodapé ou qualquer outro elemento principal da
            página.
        


        

        
Conteúdo HTML

        

            O conteúdo de uma página deve ser definido em um arquivo HTML, dentro do diretório referente à página,
            conforme colocado em views das páginas. Este arquivo, por uma questão de
            organização, é recomendável que possua como extensão ".page.html".
        

        

            O arquivo deve conter todo o conteúdo a ser inserido dentro do elemento
            <div id="route-content"></div> no arquivo src/app/app.body.html. Assim,
            pode-se construir toda uma página ou mesmo apenas seu corpo para ser encaixado em um template global, como
            descrito em template HTML.
        


        

        
CSS global

        

            Para a construção das página, existe por padrão um arquivo CSS que é aplicado globalmente a todas as telas
            da aplicação. O arquivo main.css encontra-se na raiz do projeto e contém todas as regras de estilo
            aplicadas às páginas.
        

        

            É possível, portanto, organizar todo o CSS da aplicação em um único arquivo de estilo, já pré-existente no
            projeto desde o momento de sua criação. Entretanto, também é possível especificar arquivos de CSS com
            regras específicas para cada página através dos CSS exclusivo.
        


        

        
CSS exclusivo

        

            Além do uso de um arquivo CSS global, também é possível utilizar arquivos CSS
            exclusivos para uma determinada página ao carregá-la através de uma rota. Um arquivo de CSS exclusivo
            deve ser criado junto ao HTML da página, por questões de organização, e conter a extensão
            .page.css, conforme citado em views das páginas.
        

        

            Para que o arquivo CSS seja carregado, entretanto, é necessário que ele seja referenciado no arquivo de
            setup da página. Mais informações podem ser encontradas em arquivos de setup.
        


        



        
Funções para a handler

        

            Existem também algumas poucas funções criadas para simplificar funcionalidades na hora de realizar o
            processamento em JavaScript na aplicação. Estas funções são métodos do objeto global app,
            instanciado automaticamente durante a execução do projeto.
        

        

            Tais funções podem ser chamadas e utilizadas na função handler, onde são definidas as ações JS a
            serem executadas ao renderizar uma página, conforme descrito em arquivos de setup.
        


        

        
ajax()

        

            A função realiza uma requisição HTTP a um servidor. Tal função espera
            um parâmetro, um objeto contendo ao menos os parâmetros url, que indica a URL para qual a requisição
            será feita, e success, uma função anônima usada como callback, que pode receber um parâmetro,
            o qual contém o retorno dado pelo servidor. Sua chamada é app.ajax({obj}), onde {obj}
            representa o objeto esperado como parâmetro.
        

        

            Além destes dois atributos principais, existem também outros parâmetros aceitos pela função. São eles:
            method, que indica o método HTTP utilizado na requisição (GET, POST, PUT, DELETE etc., por padrão
            utilizando GET), async, que indica se a requisição deve ser assíncrona ou não (booleano,
            por padrão true), headers, lista de chave/valor que define os headers a serem enviados na
            requisição, data, que define os dados a serem enviados como parâmetros na requisição,
            contentType, que espera o MIME type dos dados enviados na requisição, caso se aplique, e
            fail, que espera uma função callback para execução em caso de falha da
            requisição, a qual também pode esperar um parâmetro, que contém o retorno dado pelo servidor.
        

        

            Veja um exemplo de uso com todos os atributos possíveis:
        

        
			
app.ajax({

            
   url: 'server/api/function', // Essencial. URL destino da requisição

            
   method: 'post', // Opcional. Aceita os principais verbos do protocolo HTTP - por padrão, GET

            
   headers: {'Authentication': token}, // Opcional. Aceita um ou mais headers de requisição HTTP

            
   data: {'name': 'Jack'}, // Opcional. Aceita um ou mais dados para enviar na requisição

            
   contentType: 'application/json; charset=utf-8', // Opcional. Aceita o MIME type dos dados enviados

            
   async: true, // Opcional. True ou false - por padrão, true

            
   success: function(data){ // Essencial. Função de callback em caso de sucesso

            
       // JS code here

            
   },

            
   fail: function(data){ // Opcional. Função de callback em caso de falha

            
       // JS code here

            
   },

			
});

		

        

        
redirect()

        

            Realiza um redirecionamento dentro do contexto da Single Page Application, enviando o usuário a uma nova
            rota e carregando seu conteúdo. A função espera um parâmetro, uma string url, indicando qual a rota
            à qual o usuário deve ser redirecionado, e é chamada através do comando app.redirect(url),
            em que "url" representa o parâmetro esperado.
        

        

            Um simples exemplo de uso desta função:
        

        
			
app.redirect('cliente/1'); // Define a rota para onde o usuário deve ser redirecionado
